# 基于多模态大模型技术的设备检修知识检索与作业系统
## 技术方案与整体规划（v1）

> 赛题：第十五届中国软件杯 A组 · 龙芯中科
> 前提：本地 Windows 开发 → 云端大模型 API（Qwen/DeepSeek）→ 最终部署到 LoongArch + 银河麒麟高级服务器版 V11/V10
> 日期：2026-06-03（v1）｜更新：2026-06-05 —— **四大核心功能 M1–M5 已全部实现并端到端验证**，详见第 5 节路线图。

## 0. 贯穿全局的第一原则

**唯一不可妥协的硬约束：系统必须能在 LoongArch 架构 + 银河麒麟高级服务器版上运行（否则 0 分）。**

由此推导出三条选型铁律：

1. **凡是要在服务器上"运行/编译"的组件，必须选 LoongArch 生态成熟的**（后端运行时、数据库、向量检索、Web 服务器）。
2. **能"绕开" LoongArch 风险的就绕开**：
   - 前端在 Windows 上编译成静态文件，服务器只跑 Nginx 托管 → 不存在前端在 LoongArch 上构建的问题。
   - 大模型走**云端 API**（Qwen/DeepSeek）→ 服务器不需要 GPU、不需要 LoongArch 跑 PyTorch，彻底规避算力与算子兼容问题。
3. **向量检索不要引入难以在 LoongArch 编译的重型中间件（如 Milvus）**，优先用 **PostgreSQL + pgvector**（纯 C 扩展，可编译，且把"关系数据 + 向量 + 结构化过滤"统一在一个库里）。

```

└───────────────────────────┬─────────────────────────────────┘
                            │ HTTPS / REST + SSE(流式)
┌───────────────────────────▼─────────────────────────────────┐
│   Nginx (反向代理 + 静态资源托管 + SSE透传)  ←麒麟自带源     │
└───────────────────────────┬─────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────┐
│             后端服务  FastAPI (Python 3.x)                    │
│  ┌──────────┬───────────┬───────────┬──────────┬──────────┐  │
│  │ 多模态检索 │ 作业指引引擎 │ 知识沉淀/审核 │ 标注修正 │ 用户/权限 │  │
│  └──────────┴───────────┴───────────┴──────────┴──────────┘  │
│         RAG 编排层 (检索→重排→Prompt组装→LLM→引用)             │
└─────┬───────────────────────┬───────────────────┬───────────┘
      │                       │                   │
┌─────▼──────┐    ┌───────────▼─────────┐   ┌─────▼──────────┐
│ PostgreSQL │    │   对象/文件存储       │   │  云端大模型 API │
│ + pgvector │    │ (本地磁盘/MinIO)     │   │ Qwen-VL / Qwen │
│ 关系+向量+  │    │ PDF/图片/案例附件    │   │ DeepSeek       │
│ 知识图谱表  │    └─────────────────────┘   │ + Embedding API│
└────────────┘                              └────────────────┘
```

**部署形态**：B/S。服务器上只有 4 个进程：Nginx、Python(FastAPI/uvicorn+gunicorn)、PostgreSQL、(可选)MinIO。全部在 LoongArch+麒麟上有成熟支持。

## 2. 技术选型（含 LoongArch 可行性判断）

| 层 | 选型 | LoongArch/麒麟可行性 | 理由 |
|---|---|---|---|
| 前端 | **Vue3 + Vite + Element Plus + ECharts** | ✅ 不涉及（Windows 编译，服务器只放静态文件） | 生态成熟、UI 美观（对应"用户体验 20%"）；ECharts 做知识图谱/统计可视化 |
| Web 服务器 | **Nginx** | ✅ 麒麟官方源有 | 静态托管 + 反代 + SSE 流式透传 |
| 后端 | **Python 3 + FastAPI + Uvicorn/Gunicorn** | ✅ 麒麟/Loongnix 对 Python 支持好，纯 Python 依赖无架构问题 | 异步、自带 OpenAPI 文档、生态适合 AI/RAG |
| 数据库 | **PostgreSQL 14+** | ✅ 可从源码/麒麟源安装 | 稳定，支持 JSON、全文检索 |
| 向量检索 | **pgvector 扩展** | ✅ 纯 C 扩展，源码编译即可 | 免引入 Milvus；向量+结构化过滤+关系数据一库搞定 |
| 知识图谱 | **PostgreSQL 关系表建模（实体表+关系表）+ 前端 ECharts/D3 可视化**；进阶可选 Apache AGE | ✅ 避开 Neo4j 在 LoongArch 的兼容风险 | 满足"纳入知识图谱"要求，部署最稳 |
| 大模型 | **云端 API：Qwen-Max / Qwen-VL-Max / DeepSeek** | ✅ 仅需 HTTPS 出网，与架构无关 | 规避本地算力/算子问题；多模态用 Qwen-VL |
| Embedding | **云端 Embedding API（如 text-embedding-v3 / BGE 在线）** | ✅ HTTP 调用 | 避免本地跑 torch；维度统一存 pgvector |
| 文档解析 | PyMuPDF / pdfplumber（PDF）、python-docx | ✅ 纯 Python / 轻量 C | 解析摩托车维修手册等资料 |
| 鉴权 | JWT（pyjwt） | ✅ | 一线人员 / 审核员 / 管理员 角色 |

> ⚠️ **上线前必须在龙芯虚机上实测**：pgvector 源码编译、PyMuPDF 的 wheel/编译、Python 各依赖的安装。建议尽早拿到虚机跑一遍"依赖安装冒烟测试"，这是本项目最大的技术风险点。

## 3. 四大核心功能 → 技术实现映射

### 功能① 多模态知识检索（文本 / 故障图片 / 设备型号）
- **文本**：query → Embedding → pgvector 余弦相似度检索 Top-K → 重排 → LLM 生成带**引用出处**的答案（RAG）。
- **故障图片**：图片 → **Qwen-VL** 理解，输出"故障现象结构化描述" → 再走文本检索；同时可让 VL 直接结合检索片段做诊断。（跨模态：以"图生文"对齐到统一文本向量空间，最稳妥。）
- **设备型号**：作为**结构化过滤条件**（WHERE device_model=...）叠加在向量检索之上，实现"语义 + 精确过滤"混合检索。
- 关键技术点：混合检索（向量 + 关键词 BM25/全文 + 结构化过滤）、重排、引用溯源。

### 功能② 标准化作业指引
- 数据建模：`检修流程模板`（设备类型 × 检修等级 → 有序步骤）。
- 步骤化指引：前端 Stepper 组件逐步推进；每步可挂操作要点、风险提示、所需工具。
- **合规校验提醒**：步骤完成需勾选确认/必要项校验，缺项拦截。
- **个性化推送**：根据 `设备类型 + 检修等级` 动态选模板，可由 LLM 依据检索到的手册内容生成/补全流程草稿。

### 功能③ 知识沉淀与更新（含审核 + 知识图谱）
- 一线人员上传：检修案例、经验总结、图片附件 → 进入 `待审核` 队列。
- 审核员审核：通过后 → 自动**抽取实体与关系**（设备/部件/故障/原因/措施，用 LLM 做信息抽取）→ 写入知识图谱表 → 同时切块、Embedding 入向量库（实时可检索）。
- 知识图谱可视化：ECharts 关系图，点击节点查看关联案例。

### 功能④ 大模型输出标注与修正
- 每条 LLM 回答可被用户**点赞/点踩 + 文字纠正**。
- 纠正内容沉淀为"修正知识/反馈样本"，可优先注入后续相同问题的上下文（轻量"反馈增强"），形成闭环优化。
- 满足赛题"手动标注与修正大模型输出结果，优化系统适配性"。
